1 · The product — what it is & why it exists

30-second pitch. KaarigarGo (Hindi kaarigar = "craftsman/skilled worker") is an Uber-for-home-services marketplace for India. A customer opens the app, picks a category (electrician, plumber, cleaner…), and books a job; the backend finds the nearest available, KYC-verified worker ("pro"), who accepts and moves the job through en-route → arrived → in-progress → completed. Payment is cash or online (UPI QR), the platform takes a commission, the worker's wallet is credited, and both sides leave a verified review. Ops staff run a web console for KYC approvals, disputes, and payouts.

Who the system serves

  • Customers — book and track jobs, pay (UPI/cash), chat with the pro, rate them, hold a referral wallet. (mobile/customer app.)
  • Workers / pros — onboard, submit KYC, toggle online/offline, accept jobs, share live location, collect cash, request payouts. (mobile/worker app.)
  • Ops / admins — OPS_ADMIN & SUPER_ADMIN approve KYC, resolve disputes, suspend users, set commission rules, view analytics. (web Next.js console.)
Key idea: KaarigarGo is a two-sided real-time marketplace with an escrow money layer. Almost every hard part — geospatial matching, a strict booking state machine, idempotent payments, commission/settlement, and live tracking — exists because two strangers must transact money and time safely.
Q: Why phone-OTP login instead of email/password?
India is a mobile-first, phone-number-as-identity market; gig workers and customers may not have email. The User.phone column is the unique identity (@unique), email is optional. OTP also sidesteps password storage entirely — there is no password hash anywhere in the schema.

How one codebase yields two mobile apps

The two Expo apps share an identical structure and a near-identical src/ library (api.js, auth.js, theme.js, ui.js, format.js, socket.js, push.js). They diverge only in: (a) the role passed to login ('CUSTOMER' vs 'WORKER'), (b) the screen set, and (c) the worker's extra Api.createWorkerProfile / addSkill / submitKyc calls and expo-location. Each app declares its own app.json identity (app.kaarigargo.customer / app.kaarigargo.worker) and EAS project id.

Architecture note (current working tree). The mobile apps were migrated from Expo Router (file-based app/*.tsx, TypeScript) to React Navigation (a JS stack in src/navigation/stacknavigator.js). The entry is now index.js → registerRootComponent(App) → src/App.js, which mounts SafeAreaProvider → AuthProvider → NavigationContainer → StackNavigator. Key idea: the API is the contract. Mobile and web each keep a thin api client that hits the same /api/v1 envelope, so a backend change ships to all three clients without per-client glue.

5 · Backend architecture — the dominant module pattern

Every domain is a folder with the same three files: a router (index.js), a service (service.js), and Zod schemas (schemas.js). The router does HTTP plumbing only; the service does the work; the schema validates input. This is NestJS's controller/service/DTO split expressed without a framework.

The four shared primitives

  • Response envelope — every success is { data, error:null, meta } via ok(data, meta); every failure is { data:null, error:{ code, message, details }, meta:null }.
  • asyncHandler(fn) — wraps a handler so a thrown/rejected error reaches the error middleware: (req,res,next) => Promise.resolve(fn(...)).catch(next).
  • validate(schema, value) — Zod parse; on failure throws HttpError(400,'Validation failed', issues).
  • HttpError + helpers — BadRequest/Unauthorized/Forbidden/NotFound/Conflict carry an HTTP status; the global handler maps 5xx to a generic message (no internal leakage) and passes 4xx through with details.
// the canonical handler shape — bookings/index.js
router.post('/',
  requireRoles('CUSTOMER'),
  asyncHandler(async (req, res) => {
    const dto = validate(createBookingSchema, req.body);          // zod
    res.status(201).json(ok(await bookings.create(req.user.id, dto)));  // envelope
  }),
);

Router mount order matters

server/app.js mounts ~25 routers under one /api/v1 router. Order is deliberate: specific paths before general ones. /worker/profile and /worker/skills are mounted before /worker (payouts), and the catch-all /admin router is mounted last among /admin* so /admin/kyc, /admin/disputes, /admin/catalog, etc. win first.

Why a global response envelope? Clients parse one shape forever. The mobile api() helper reads json.error to decide success and throws ApiError(code, message) uniformly; a 401 with auth:true transparently triggers the refresh-and-retry path. Without the envelope every endpoint would need bespoke error parsing on three clients.
Q: How does an unexpected exception not crash the server or leak a stack trace?
asyncHandler funnels every rejection into errorHandler. There, status>=500 is console.error'd server-side but returned to the client as a flat { code:500, message:'Internal server error' } — no stack, no SQL. 4xx errors (your own HttpErrors) pass their message and details through because they're safe, intentional signals.

6 · Authentication & authorization, end-to-end

Auth is phone OTP → JWT access + rotating refresh. There are no passwords. All four /auth/* routes are public; everything else is gated by middleware.

Step 1 — OTP issuance (Redis-backed, auth/otp.js)

  • A 6-digit code via crypto.randomInt(0, 1_000_000), zero-padded.
  • Stored hashed: sha256(`${phone}:${code}:${pepper}`) where the pepper is JWT_ACCESS_SECRET — the raw code never sits in Redis.
  • Guards: 30 s cooldown between requests, 5 codes/hour cap, 5 verify attempts then the code is deleted, 300 s TTL.

Step 2 — verify & token issue (auth/auth.service.js)

async function verifyOtp(phone, code, role) {
  await otp.verify(phone, code);                    // throws on bad/expired code
  let user = await prisma.user.findUnique({ where: { phone } });
  if (!user) user = await prisma.user.create({ data: { phone, role: role ?? 'CUSTOMER' } });
  if (user.status === 'SUSPENDED') throw new HttpError(401, 'Account suspended');
  const tokens = await issueTokens(user.id, user.role);
  return { user: publicUser(user), isNew, ...tokens };
}

First successful OTP for a new phone auto-creates the user with the requested role — login and signup are the same call.

Step 3 — token model

TokenClaimsTTLStored where
access{ sub, role, type:'access' }900 snowhere (stateless)
refresh{ sub, jti, type:'refresh' }30 dRedis key refresh:<uid>:<jti> + set refresh:jtis:<uid>

Step 4 — refresh rotation

refresh() verifies the JWT, checks the jti still exists in Redis, then deletes it before issuing a new pair — a one-time-use rotating refresh token. A stolen-and-replayed refresh token fails because its jti was already consumed. logout(token, allDevices) deletes one jti or the whole per-user set; it's idempotent (an already-invalid token still "succeeds").

The middleware (middleware/auth.js)

  • requireAuth — demands Bearer access token, sets req.user = { id, role }; rejects non-access tokens.
  • optionalAuth — attaches req.user if present, never throws (public-but-personalised reads).
  • requireRoles(...roles) — 403 unless req.user.role is in the list.
Key idea: access tokens are stateless and short; revocation power lives in Redis on the refresh side. You get cheap, horizontally-scalable auth checks (no DB hit per request) and still the ability to kill sessions.
Q: The pepper for OTP hashing is JWT_ACCESS_SECRET. Is that a problem?
It's a pragmatic reuse: it keeps OTP hashes non-reversible without a separate secret to manage. The risk is blast-radius coupling — rotating the JWT secret invalidates in-flight OTPs (a 5-minute annoyance, not a breach). A dedicated OTP_PEPPER would be cleaner; the trade-off was fewer env vars for the demo.
Why store the code hashed, not plaintext, in Redis with a 5-min TTL anyway? Defence in depth. A Redis snapshot/dump or an accidental KEYS * leak would otherwise expose live login codes for every number mid-flight. Hashing means a leaked Redis still can't be replayed without brute-forcing 10⁶ space per phone within the TTL.

7 · Roles, scoping & access control

Four roles (UserRole): CUSTOMER, WORKER, OPS_ADMIN, SUPER_ADMIN. Authorization is enforced at three layers.

Layer 1 — router-level role gates

Whole routers pin their audience: router.use(requireAuth, requireRoles('WORKER')) in worker/profile, worker/skills, payments/payouts; requireRoles('OPS_ADMIN','SUPER_ADMIN') in admin. Some methods tighten further — POST /admin/users/:id/role adds an inline requireRoles('SUPER_ADMIN').

Layer 2 — resource ownership (the real ACL)

Role isn't enough; you must own the row. bookings/service.js centralises this in assertParticipant(bookingId, user): the caller must be the booking's customer, the assigned worker's user, or an admin — else Forbidden. Worker mutations additionally require b.workerId === wp.id ("Not assigned to you").

async function assertParticipant(bookingId, user) {
  const b = await prisma.booking.findUnique({ where:{id:bookingId},
    include:{ worker:{ select:{ userId:true }}}});
  if (!b) throw NotFound('Booking not found');
  const isCustomer = b.customerId === user.id;
  const isWorker   = b.worker && b.worker.userId === user.id;
  const isAdmin    = user.role === 'OPS_ADMIN' || user.role === 'SUPER_ADMIN';
  if (!isCustomer && !isWorker && !isAdmin) throw Forbidden('Not your booking');
  return b;
}

Layer 3 — the realtime gateway re-checks

Socket events don't trust the room name. booking.join, chat.send, and location.ping each call isParticipant(bookingId, userId) against the DB before acting, and location.ping additionally requires socket.data.role === 'WORKER'.

Why "secure-by-default"? The comment in middleware/auth.js spells out the rule: apply requireAuth on every protected router; public routes simply omit it. There is no global "allow all and blocklist" — a new router is closed until you mount auth on it, so forgetting a guard fails safe (401), not open.

Q: Is this multi-tenant?
Not org-multi-tenant — it's a single marketplace. "Tenancy" here is per-user data scoping: every list query is filtered by customerId/workerId derived from req.user (e.g. listForUser branches on role and filters to the caller's own bookings). The isolation boundary is the user, enforced by ownership checks, not a tenant id column.

9 · Geospatial discovery & matching (PostGIS)

"Find the nearest available pro" is a single PostGIS query in workers/discovery.js. Two entry points:

findBestWorkerId(categoryId, lat, lng) — auto-assign at booking time

SELECT wp.id AS "workerId"
  FROM worker_profiles wp
  JOIN worker_skills ws ON ws."workerId" = wp.id AND ws."categoryId" = $3
 WHERE wp."availabilityStatus" = 'ONLINE'
   AND wp."kycStatus" = 'APPROVED'
   AND wp.base_location IS NOT NULL
   AND ST_DWithin(wp.base_location,
        ST_SetSRID(ST_MakePoint($1,$2),4326)::geography,
        wp."serviceRadiusKm" * 1000)              -- radius in metres
 ORDER BY ST_Distance(wp.base_location, ST_SetSRID(ST_MakePoint($1,$2),4326)::geography) ASC,
          wp."ratingAvg" DESC, wp."reliabilityScore" DESC
 LIMIT 1

It filters to pros who are online + KYC-approved, whose service radius covers the customer (ST_DWithin uses each worker's own serviceRadiusKm), then ranks by distance → rating → reliability. search(...) is the browse variant: KYC-approved within radius, returns the ranked list with a computed distanceM.

Why PostGIS instead of "compute haversine in Node"? The radius predicate must run inside the query so the database can index and prune — pulling every worker into Node to measure distance is O(all workers) per booking. ST_DWithin on a geography column with a GiST index lets Postgres discard far workers before sorting, and "radius is per-worker" can't be expressed as a single client-side bounding box.
Q: BigInt/Numeric from raw SQL — any catch?
Yes. $queryRawUnsafe returns Postgres numeric/bigint as JS objects, so discovery explicitly coerces with Number(r.ratingAvg), Number(r.distanceM), etc. before returning JSON — otherwise the client gets {"$bigint":"…"} or serialization errors.
Caveat: auto-assign requires the worker to be ONLINE; search/browse only requires APPROVED. An instant booking with no online pro in range gets workerId=null and stays REQUESTED until matched — the customer UI shows "Finding the best nearby pro…".

10 · The money system — commission, settlement, wallet, payouts

All amounts are integer paise. The flow: customer pays (online escrow or cash) → platform computes commission → worker wallet credited → worker requests payout.

Commission resolution (money/commission.js)

Strict precedence, newest effective rule wins: CATEGORY rule → GLOBAL rule → category.defaultCommissionPct → 15%. Result is clamped: max(0, min(commission, amount)) so it can never exceed the job value.

const commission = Math.round((amountPaise * pct) / 100) + fixed;
return Math.max(0, Math.min(commission, amountPaise));

The wallet ledger (money/wallet.js)

Append-only with a stored runningBalance. record(tx, walletId, type, signedAmount, reference) must run inside a Prisma $transaction: it reads current balance, writes a WalletTransaction (storing Math.abs(amount) + sign-derived runningBalance), and updates Wallet.balance atomically.

Settlement (money/settlement.js) — two flows

FlowTriggerWhat happens
Online (escrow)settleIfReady fires on both COMPLETED and PAID (whichever lands last)In one transaction: credit worker wallet amount − commission, set booking SETTLED + commissionAmount, write a settled event. Then emit + notify.
CashsettleCash when worker taps "collect cash" on a COMPLETED cash jobRecord a synthetic PAID Payment(method:CASH), then debit the worker's wallet by the commission owed (worker already holds the full cash), mark SETTLED.
Key idea — settlement is order-independent. An online job can be paid before it's completed or completed before it's paid. settleIfReady is called from both the payment path (applyPayment) and the completion path (updateStatus), and it no-ops unless the booking is simultaneously COMPLETED and has a PAID payment. Idempotent: a second call sees status==='SETTLED' and returns alreadySettled.

Payouts (payments/payouts.js)

POST /worker/payouts/request validates amount ≤ balance, then in a transaction debits the wallet (PAYOUT) and creates a Payout(QUEUED). GET /worker/earnings aggregates wallet balance, lifetime credits, and pending payout totals.

Why model cash as a wallet debit? In cash mode the worker physically holds 100% of the money, but the platform is still owed commission. Crediting the worker would double-count; instead the ledger records a negative COMMISSION entry, so the wallet reflects "you owe the platform ₹X" and future online earnings net it off. This keeps one consistent ledger for both payment modes.

11 · Payments & the UPI flow

Provider abstraction (payments/provider.js)

The module exports razorpay if RAZORPAY_KEY_ID is set, else a mock — chosen once at require time. Both expose createOrder() and verifySignature(). Razorpay's verifySignature is an HMAC-SHA256 of the raw request body against RAZORPAY_WEBHOOK_SECRET.

Three ways a booking gets paid

  1. Gateway order — createOrder creates a PENDING Payment + provider order; the real money event arrives via POST /payments/webhook (HMAC-verified) → applyPayment.
  2. Dev mock-pay — POST /payments/order/:orderId/mock-pay (blocked in production) flips the order to PAID for testing.
  3. UPI QR — createUpiQr renders a upi://pay?pa=&pn=&am=&cu=INR&tn=&tr= deep link as a data-URL QR; the customer pays in any UPI app, then either party calls POST /payments/upi/confirm to mark it PAID.

Idempotency is the whole game (applyPayment)

async function applyPayment(p) {
  const payment = await prisma.payment.findFirst({ where:{ providerOrderId: p.providerOrderId }});
  if (!payment) return { ignored:true, reason:'unknown_order' };
  if (payment.status === 'PAID') return { ok:true, idempotent:true };   // ← double-fire guard
  await prisma.payment.update({ where:{id:payment.id}, data:{ status:p.status, providerPaymentId:p.providerPaymentId, method:mapMethod(p.method) }});
  const booking = await prisma.booking.findUnique({ where:{id:payment.bookingId}});
  if (booking) realtime.emitToUser(booking.customerId, 'payment.updated', { bookingId:booking.id, status:p.status });
  return { ok:true, paymentStatus:p.status, settlement: p.status==='PAID' ? await settlement.settleIfReady(payment.bookingId) : null };
}
Key idea: webhooks fire more than once. Every payment mutation keys on providerOrderId and short-circuits if already PAID, so duplicate webhooks, a double "I've paid" tap, and a retried mock-pay all converge to one settlement.
Q: Why capture the raw body in app.js if you already have parsed JSON?
HMAC must be computed over the exact bytes the provider signed, not a re-serialized object (key order/whitespace would differ and break the signature). express.json({ verify:(req,_res,buf)=>{ req.rawBody = buf; }}) stashes the raw Buffer so verifySignature(req.rawBody, signature) can validate the webhook.
Caveat — UPI-QR "confirm" is trust-based. Without a gateway callback, confirmUpi believes the human that money arrived. That's why it's allowed for either the customer or the assigned worker, and why it's idempotent. For real money you'd reconcile against bank/UPI settlement reports; here it's the pragmatic zero-integration demo path (added in the "payment fixes" commits — see Part 22).

12 · Real-time — Socket.IO rooms & events

Attached in server.js via require('./realtime').attach(server). Auth happens in the connection handler: the socket must present a valid access JWT in handshake.auth.token (or query) or it's disconnected. On connect the socket auto-joins user:<id>.

Rooms

Q: A socket joined booking:42. Can it forge events for booking 99?
No. The room name is never trusted — every handler re-runs isParticipant(data.bookingId, socket.data.userId) against the database using the userId from the verified JWT, not from the payload. Joining a room you don't belong to is rejected, and even a stale membership can't emit cross-booking.

13 · Notifications & push

notifications/index.js exposes one function, notify(userId, type, title, body, data), used everywhere a human should hear about something (job accepted, on the way, completed, you got paid, new job request).

async function notify(userId, type, title, body, data) {
  await prisma.notification.create({ data:{ userId, channel:'PUSH', type, title, body, data }}).catch(()=>undefined);
  const devices = await prisma.device.findMany({ where:{ userId }});
  const messages = devices
    .filter(d => /^Expo(nent)?PushToken\[/.test(d.fcmToken))   // only real Expo tokens
    .map(d => ({ to:d.fcmToken, title, body, data: data ?? {}, sound:'default' }));
  if (messages.length === 0) return;
  try { await fetch('https://exp.host/--/api/v2/push/send', { method:'POST', /* … */ }); }
  catch (e) { console.warn(`Push send failed: ${e.message}`); }
}

It does two things: persists an in-app Notification row (so the bell icon works even with no device) and fans out an Expo push to every registered device whose token looks like a real ExponentPushToken[...]. Devices register via POST /me/devices (an upsert on the unique fcmToken).

Gotcha — graceful degradation everywhere. The notification create is .catch(()=>undefined) and the push fetch is wrapped in try/catch. A failed push or a missing device never breaks the business transaction that triggered it — settlement still completes even if Expo is down. The mobile registerForPush similarly no-ops in Expo Go / on simulators.
Q: The env has FIREBASE_* but pushes go through Expo. Why?
Expo's push service (exp.host) brokers to FCM/APNs for you, so the app only deals with ExponentPushToken[...] values and the server posts to one endpoint. The FIREBASE_*/SMS_*/WHATSAPP_* env slots are placeholders for swapping in direct providers later — the NotificationChannel enum already models PUSH/SMS/WHATSAPP/INAPP.

14 · The booking lifecycle — the state machine

The BookingStatus enum has 11 states; legal worker transitions are a tiny table in bookings/status.js. Everything else (cancel, settle) is governed by explicit guards in the service.

const WORKER_TRANSITIONS = {            // bookings/status.js
  ACCEPTED:    ['EN_ROUTE'],
  EN_ROUTE:    ['IN_PROGRESS'],
  IN_PROGRESS: ['COMPLETED'],
};
const TERMINAL_STATUSES = ['COMPLETED','SETTLED','REJECTED',
  'CANCELLED_BY_CUSTOMER','CANCELLED_BY_WORKER','EXPIRED'];
 REQUESTED ──accept──▶ ACCEPTED ──▶ EN_ROUTE ──▶ IN_PROGRESS ──▶ COMPLETED ──settle──▶ SETTLED
     │                    │            │                                                  ▲
     ├─reject─▶ REJECTED  └─ cancel ───┴─ (worker: ACCEPTED/EN_ROUTE only) ─▶ CANCELLED_BY_WORKER
     └─ cancel (customer: REQUESTED/ACCEPTED) ─▶ CANCELLED_BY_CUSTOMER
   online: settleIfReady needs COMPLETED + PAID  ·  cash: worker "collect cash" ⇒ settleCash ⇒ SETTLED

What each transition does (side-effects)

  • create — resolves a worker (explicit or nearest-online), writes the booking + geo point, emits booking.worker_assigned + push to the pro.
  • accept/reject — worker-only, must be REQUESTED; reject docks reliabilityScore −3.
  • updateStatus — worker-only, gated by canWorkerTransition; on COMPLETED sets finalPrice, increments completedJobs, reliability +1, then settleIfReady.
  • cancel — role-aware windows: customer can cancel from REQUESTED/ACCEPTED; worker only from ACCEPTED/EN_ROUTE (and takes reliability −5).
  • track / cashConfirm / sos — live location, cash settlement, and a safety broadcast to the booking room + event log.
Key idea: the worker can only move the job forward one step at a time. An illegal jump (e.g. ACCEPTED→COMPLETED, skipping work) throws Conflict('Illegal transition …'). The state machine is the single source of truth for "what can happen next", and every move appends a BookingEvent so the timeline is reconstructable.
Q: Two workers — can both accept the same request?
No. A booking is auto-assigned a single workerId at creation, and accept checks b.workerId === wp.id ("Not assigned to you") and b.status === 'REQUESTED' ("Cannot accept from …"). The first accept flips it to ACCEPTED; any second attempt hits the status guard and is rejected with 409.

15 · Feature-by-feature walkthrough

KYC (kyc/service.js)

Workers submit document references (docType, fileUrl) — the system stores references only, not files. Submitting flips kycStatus NONE/REJECTED → PENDING. Ops approve/reject per document; approval sets the profile APPROVED and emits kyc.updated; rejection sets REJECTED only if no doc is approved. KYC gates earning power: worker/profile PATCH refuses availabilityStatus:'ONLINE' unless kycStatus==='APPROVED', and discovery only matches approved pros.

Reviews (reviews/service.js)

Only after COMPLETED/SETTLED, only by a participant, one per reviewer per booking (duplicate → 409). Reviews are isVerified:true because they're tied to a real booking. A customer's review of a worker triggers recompute, which re-aggregates ratingAvg/ratingCount onto the WorkerProfile (denormalised for fast discovery sorting).

Referrals (referrals/service.js)

Each user has one share code (generateCode derives a short base-36 code from their UUID). Applying a valid code credits ₹50 (5000 paise) to both parties' CUSTOMER wallets in one transaction. Guards: can't use your own code, one code per account (else 409).

Disputes & chat

Either party can raise a dispute on a booking (OPEN → ops UNDER_REVIEW → RESOLVED) with evidence URLs. Chat is per-booking, cursor-paginated, with read receipts via markRead (marks the other party's unread messages read).

Admin console API (admin/*)

OPS_ADMIN/SUPER_ADMIN only: analytics overview, user search + suspend/reinstate, SUPER_ADMIN-only role changes, worker feature-flagging, booking search, audit-log reads, plus dedicated /admin/kyc, /admin/disputes, /admin/catalog, /admin/commission-rules, /admin/payouts routers.

Key idea: denormalisation is deliberate. ratingAvg, completedJobs, reliabilityScore, and commissionAmount are precomputed onto rows so the hot paths (discovery ranking, settlement) never aggregate at request time.

20 · Rapid-fire interview question bank

Q: Why integer paise for money?
Floats can't represent 0.1 exactly; summing them drifts. Int paise is exact; format to rupees only at the edge (rupees() divides by 100).
Q: Access token expired but user is active — what happens?
Client gets 401, silently refreshes (rotating refresh), retries once. Server never tracks access tokens.
Q: How is "nearest worker" computed?
PostGIS ST_DWithin (radius) + ST_Distance (sort) in one indexed SQL query, filtered to ONLINE+APPROVED pros who serve the category.
Q: Online booking — does payment or completion settle it?
Whichever is last. settleIfReady runs from both paths and only acts when COMPLETED and PAID; it's idempotent.
Q: Cash job — how does the platform get its cut?
Worker holds the cash; settleCash records a negative COMMISSION wallet entry, so the worker "owes" the platform, netted against future earnings.
Q: Webhook fires twice — double charge?
No. applyPayment keys on providerOrderId and returns idempotent if already PAID.
Q: Why capture req.rawBody?
HMAC must be over the exact signed bytes; re-serialized JSON would break the signature.
Q: Can a worker go online without KYC?
No — PATCH profile rejects ONLINE unless kycStatus==='APPROVED'; discovery also requires APPROVED.
Q: Why Redis for OTP and refresh?
Both are ephemeral with natural TTLs and need fast atomic ops (counters, set membership); Postgres would be the wrong tool.
Q: How do you stop a user reading another's booking?
assertParticipant on REST and isParticipant on sockets — ownership derived from the JWT, not the payload.
Q: Where does authorization actually live?
Three layers: requireRoles (router), assertParticipant/workerId checks (service), isParticipant (gateway).
Q: Why plain JS Express over the original NestJS?
Delete the build step + DI overhead; faster cold starts on free hosting; same controller/service/DTO shape expressed as router/service/schema files.
Q: Why two mobile apps from one repo?
Distinct store listings + smaller bundles + disjoint permissions; shared src/ libraries avoid duplicating API/auth/socket logic.
Q: First thing that breaks at 2 API instances?
In-memory Socket.IO rooms — add the Redis adapter.

22.2 · "The app's icons are all blank squares" (commit 8fe913a)

Q: Ionicons render in Expo Go but every icon is a blank box in the built APK. Why?
The icon font wasn't bundled into the native build, and nothing loaded it. The fix embeds Ionicons.ttf in the Android assets and kicks off a non-blocking font load.
Before — the root layout never loaded the font; the release build shipped without the ttf, so glyphs fell back to nothing.
After — bundle the ttf + load it without blocking the UI: 
+import { Ionicons } from '@expo/vector-icons';
+import { useFonts } from 'expo-font';
 export default function RootLayout() {
+  // Icon font is embedded natively (android/app/src/main/assets/fonts/Ionicons.ttf).
+  // Kick off a background load too, but never block the UI on it.
+  useFonts(Ionicons.font);
Two subtleties (per the build notes): the asset must be lowercase ionicons.ttf in release builds (Android asset lookup is case-sensitive and @expo/vector-icons requests family ionicons), and the load must stay non-blocking — gating render on it hangs the whole app if the font ever fails. After the React-Navigation migration the same non-blocking useFonts(Ionicons.font) lives in src/App.js.

22.3 · "Paying did nothing useful" (commits a0f2198 + 9336309)

Q: The old "Pay" button just created a mock order. How do customers actually pay now?

Before — the booking screen's pay() just hit the order/mock-pay endpoints and showed a toast; no real UPI:

}

22.4 · "It works on my machine but the phone can't reach the API" (commit c8b27ab)

Q: A real device build pointed at localhost — why did nothing load?
localhost on a phone is the phone itself, not the dev machine. The clients were repointed at the deployed Render API.
Before — mobile app.json baked in a host-only URL, and EAS pointed at a domain that wasn't live: 
"extra": { "apiUrl": "http://localhost:3000/api/v1" }
// eas.json:  "EXPO_PUBLIC_API_URL": "https://api.kaarigargo.app/api/v1"
After — point web + both mobile apps at the live Render host: 
"extra": { "apiUrl": "https://kaarigargo-api.onrender.com/api/v1" }
// eas.json (preview + production):
"EXPO_PUBLIC_API_URL": "https://kaarigargo-api.onrender.com/api/v1"
This is why the API client resolves its base as process.env.EXPO_PUBLIC_API_URL ?? Constants.expoConfig?.extra?.apiUrl ?? 'http://localhost…' — env wins for EAS builds, app.json is the baked fallback, localhost only for the simulator.
Pattern across all four: every fix is a demo-readiness fix — making a feature work end-to-end without a paid dependency (SMS gateway, payment gateway, custom domain) or a native build quirk (font bundling). Each is small, surgical, and left a comment explaining the trade-off and how to harden it for real users.
